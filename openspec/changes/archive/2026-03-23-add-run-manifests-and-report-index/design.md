# Design

## Current State

The current artifact flow is organized around raw execution output:

- `packages/cli/src/testRunner.ts` creates a run directory and delegates file output to `ReportWriter`
- `packages/cli/src/reportWriter.ts` writes:
  - `summary.json`
  - per-spec `result.json`
  - per-step JSON files
  - screenshots
  - copied recordings
  - one single-run `index.html`
- `packages/cli/src/reportTemplate.ts` renders the detail page directly from the in-memory `summary` and `specs` objects passed at finalize time

This is enough for one report page, but not enough for a report system.

Missing pieces:

- no root run-history manifest
- no root run-history HTML page
- no stable single-run manifest that captures authored inputs, safe env context, and artifact paths in one place
- no clean renderer contract for a simple static server or future hosted UI

## Decision

Adopt a two-level manifest model:

- `run.json` is the canonical source of truth for a single run detail page
- `runs.json` is the canonical source of truth for the run history page

Everything else becomes either:

- raw drill-down artifacts
- copied media assets
- compatibility files kept for existing workflows

This gives us a clear rendering contract:

```text
runs.json         -> render run list page
<run-id>/run.json -> render single run detail page
other files       -> passive assets linked by relative path
```

## Sources Of Truth

### Single Run

`<run-id>/run.json` should be sufficient to render the full detail page without:

- scanning directories
- reopening workspace YAML files
- reading per-spec `result.json`
- reading `summary.json`

The HTML page can still link to raw files, but it should not depend on them for structure.

### Run History

`.finalrun/artifacts/runs.json` should be sufficient to render the run list page without:

- scanning every run directory from the browser
- opening each run manifest individually

`runs.json` should be regenerated by CLI code after each completed run by scanning available `run.json` files. That keeps the write path simple and makes the index rebuildable.

## Artifact Layout

Recommended layout:

```text
.finalrun/artifacts/
  runs.json
  index.html
  <run-id>/
    run.json
    summary.json
    index.html
    runner.log
    input/
      run-context.json
      env.snapshot.yaml
      env.json
      specs/
        <spec-id>.yaml
        <spec-id>.json
    tests/
      <spec-id>/
        result.json
        recording.mp4
        screenshots/
          001.jpg
        steps/
          001.json
```

Rules:

- all paths stored in manifests are relative paths
- every copied snapshot is immutable once the run is finalized
- secret values are never written into snapshot YAML, JSON manifests, or HTML

## `<run-id>/run.json`

Recommended top-level sections:

- `schemaVersion`
- `run`
- `input`
- `specs`
- `paths`

### `run`

Contains run-wide metadata:

- `runId`
- `status: 'success' | 'failure'`
- `failurePhase: 'validation' | 'setup' | 'execution' | 'finalization' | null`
- `startedAt`, `completedAt`, `durationMs`
- `envName`
- `platform`
- `model: { provider, modelName, label }`
- `app: { source: 'repo' | 'override', label, overridePath? }`
- `selectors: string[]`
- `counts`
  - `specs: { total, passed, failed }`
  - `steps: { total, passed, failed }`
- `firstFailure`
- `diagnosticsSummary?`

### `input`

Contains authored inputs and safe context:

- `environment`
  - `envName`
  - `workspaceEnvPath?`
  - `snapshotYamlPath?`
  - `snapshotJsonPath`
  - `variables`
  - `secretReferences: Array<{ key, envVar }>`
- `specs`
  - `specId`
  - `specName`
  - `relativePath`
  - `workspaceSourcePath`
  - `snapshotYamlPath`
  - `snapshotJsonPath`
- `cli`
  - `command`
  - `requestedPlatform?`
  - `appOverridePath?`
  - `debug`
  - `maxIterations`

### `specs`

Each spec entry should be rich enough to render the detail page directly.

Recommended shape per spec:

- identity
  - `specId`
  - `specName`
  - `relativePath`
  - `workspaceSourcePath`
- authored input
  - `snapshotYamlPath`
  - `snapshotJsonPath`
  - `authored`
    - `description`
    - `preconditions[]`
    - `setup[]`
    - `steps[]`
    - `assertions[]`
  - `bindingReferences`
    - `variables[]`
    - `secrets[]`
- effective input
  - `effectiveGoal`
- result
  - `status`
  - `message`
  - `analysis`
  - `startedAt`
  - `completedAt`
  - `durationMs`
  - `counts: { executionStepsTotal, executionStepsPassed, executionStepsFailed }`
  - `firstFailure?`
- media
  - `recordingPath?`
  - `previewScreenshotPath?`
- paths
  - `resultJsonPath`
  - `stepsDirPath`
  - `screenshotsDirPath`
- `executionSteps[]`

Each execution step should include:

- `stepNumber`
- `iteration`
- `actionType`
- `naturalLanguageAction`
- `reason`
- `analysis`
- `status`
- `errorMessage`
- `timestamp`
- `durationMs`
- `videoOffsetMs`
- `authoredRef?`
  - `section: 'setup' | 'steps' | 'assertions'`
  - `index`
- `paths`
  - `stepJsonPath`
  - `screenshotPath?`
- `thought`
- `actionPayload`
- `trace`
- `timing`

## Input Snapshot Rules

### YAML Specs

For every selected YAML file, copy the exact source used at run time into:

- `input/specs/<spec-id>.yaml`

Also write a parsed JSON companion:

- `input/specs/<spec-id>.json`

The parsed JSON removes the need for the renderer to parse YAML and gives the UI stable access to:

- name
- description
- preconditions
- setup
- steps
- assertions
- referenced variables
- referenced secrets

### Environment Snapshot

Write two environment artifacts:

- `input/env.snapshot.yaml` containing the safe env config view with variable values and secret placeholders such as `${FINALRUN_EMAIL}`
- `input/env.json` containing:
  - `envName`
  - `workspaceEnvPath`
  - `variables`
  - `secretReferences`

Do not write `RuntimeBindings.secrets` anywhere.

## UI Model

## Run List Page

The root page should answer:

- which runs failed
- how many specs passed or failed
- what environment and platform were used
- what failed first
- which run to open next

Recommended layout:

```text
┌──────────────────────────────────────────────────────────────┐
│ FinalRun Reports                                            │
│ [All] [Failed] [Passed] [android] [ios] [search]            │
├──────────────────────────────────────────────────────────────┤
│ Failed  staging  android  3/5 passed  2m14s  auth/login     │
│ Passed  dev      ios      8/8 passed  4m01s  no failures    │
│ Failed  qa       android  0/1 passed  0m19s  setup failure  │
└──────────────────────────────────────────────────────────────┘
```

Each row or card should show:

- status chip
- started time
- env
- platform
- model label
- `passed/failed/total`
- duration
- first failure summary
- preview screenshot when available
- link to detail page
- quick links to log and raw manifest

## Single Run Detail Page

The detail page should answer:

- what the author wrote
- what environment/context was used
- what the model effectively received
- what happened step by step

Recommended structure:

```text
┌ Header: status, env, platform, model, duration, counts
├ Tabs: Overview | Inputs | Specs | Logs
│
├ Overview
│  ├ failed specs first
│  ├ first failure summary
│  └ spec duration / pass-fail table
│
├ Inputs
│  ├ env snapshot
│  ├ variables
│  ├ secret references
│  └ selected YAML snapshots
│
├ Specs
│  ├ left: spec list
│  └ right:
│     ├ authored YAML / parsed sections
│     ├ effective goal text
│     ├ recording player
│     ├ selected screenshot
│     ├ execution timeline
│     └ raw artifact links
│
└ Logs
   └ runner.log plus failure diagnostics
```

Important presentation rules:

- failed specs should be surfaced before passed specs
- authored input and execution output should be visible side by side
- the page should make it obvious when a failure happened before any recorded agent action
- raw links should exist, but the main UI should not force the user to open raw JSON just to understand the run

## Architecture Changes

## Common Models

Extend `packages/common/src/models/RunArtifacts.ts` to add:

- `RunManifestRecord`
- `RunManifestSpecRecord`
- `RunManifestStepRecord`
- `RunIndexRecord`
- `RunIndexEntryRecord`
- `FailurePhase`

Keep existing `RunSummaryRecord`, `SpecArtifactRecord`, and `StepArtifactRecord` for compatibility and raw drill-down output.

## CLI Report Writing

Refactor `packages/cli/src/reportWriter.ts` into two responsibilities:

- low-level artifact writing
  - screenshots
  - step JSON
  - spec result JSON
  - copied recordings
  - log file
- manifest assembly
  - input snapshots
  - run manifest
  - summary

Recommended helper split:

- `reportWriter.ts`
  - owns run directory lifecycle
  - writes raw artifacts
- `runManifestBuilder.ts`
  - assembles render-ready `run.json`
- `runIndexWriter.ts`
  - scans `run.json` files and rebuilds `runs.json`
- `runDetailTemplate.ts`
  - renders single-run HTML from `RunManifestRecord`
- `runIndexTemplate.ts`
  - renders run-list HTML from `RunIndexRecord`

## CLI Read/Serve Surfaces

Add small read-side helpers in `packages/cli/src`:

- `runHistory.ts`
  - loads `runs.json`
  - falls back to rebuild-from-scan when needed
- `reportServer.ts`
  - serves `.finalrun/artifacts` as static files

Update `packages/cli/bin/finalrun.ts` with:

- `finalrun runs`
- `finalrun runs --json`
- `finalrun report serve [--port <n>]`

`finalrun test` should also print:

- the per-run report path
- the root run-history page path

## Test Runner Flow

Recommended `finalrun test` flow after this change:

```text
runCheck()
  -> resolve workspace/env/specs/app override
  -> create ReportWriter
  -> write input snapshots
  -> prepareGoalSession()
  -> execute each spec
  -> write raw spec artifacts
  -> build run.json
  -> rebuild runs.json
  -> render <run-id>/index.html
  -> render artifacts/index.html
```

Illustrative code shape:

```ts
const checked = await runCheck(...);

const { reportWriter, runDir } = await createReportWriter({
  workspace: checked.workspace,
  runContext: {
    envName: checked.environment.envName,
    selectors: normalizedSelectors,
    provider: model.provider,
    modelName: model.modelName,
    requestedPlatform: options.platform,
    appOverridePath: checked.appOverride?.appPath,
    startedAt,
  },
});

await reportWriter.writeRunInputs({
  specs: checked.specs,
  environment: checked.environment,
});

const goalSession = await prepareGoalSession(...);

for (const spec of checked.specs) {
  const goal = compileSpecToGoal(spec, checked.environment.bindings);
  const goalResult = await executeGoalOnSession(...);
  await reportWriter.writeSpecRecord(spec, goalResult, checked.environment.bindings, goal);
}

await reportWriter.finalize({
  actualPlatform: goalSession.platform,
  failurePhase: encounteredFailure ? 'execution' : null,
});

await rebuildRunsIndex(checked.workspace.artifactsDir);
```

### Failure Runs

Failure runs should still produce renderable manifests.

Rules:

- validation failure:
  - write a run entry with `failurePhase: 'validation'`
  - spec list may be empty
- setup failure after `runCheck`:
  - snapshot selected specs and env
  - write `failurePhase: 'setup'`
- execution failure:
  - include completed spec records plus the failed spec record
  - write `failurePhase: 'execution'`

This keeps both pages useful even when no successful spec completed.

## Risks

- snapshotting more input data increases artifact size, so manifests should reference raw YAML files instead of embedding large raw blocks repeatedly
- root index rebuilding must be tolerant of partially-written or corrupted run folders
- secret handling must remain strict when copying env context and action payload text
- the detail page can become noisy if it tries to show every raw field at once, so UI defaults should prioritize summary first and raw links second

## Resolved Decisions

1. `run.json` is the only required JSON for rendering a single run detail page.
2. `runs.json` is the only required JSON for rendering the run history page.
3. existing `summary.json`, `result.json`, and step JSON files remain for compatibility and debugging.
4. input YAML and safe env metadata are copied into the run bundle so old runs remain stable even if the workspace changes later.
5. resolved secret values are never stored in manifests or snapshots.
6. `runs.json` is derived and rebuildable; `run.json` is the per-run source of truth.
